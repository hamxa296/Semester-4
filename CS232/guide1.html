PART 1 — PostgreSQL + psycopg2 CRUD

First: The Full Flow

In Python with psycopg2, CRUD means:

OperationSQL
CreateINSERT
ReadSELECT
UpdateUPDATE
DeleteDELETE

The typical connection and execution flow is:

import psycopg2

conn = psycopg2.connect(
    host="localhost",
    database="testdb",
    user="postgres",
    password="1234"
)

cur = conn.cursor()

# query execute
cur.execute("SQL QUERY")

conn.commit()

cur.close()
conn.close()

Important Theory Concepts

Your teacher will absolutely ask these.

1. execute() DOES NOT PRINT RESULTS

WRONG: cur.execute("SELECT * FROM students")
This executes successfully, but it DOES NOT show rows.

CORRECT:
rows = cur.fetchall()
print(rows)

2. SELECT DOES NOT NEED commit()

This is a classic trap:
cur.execute("SELECT * FROM students")
conn.commit()
This STILL works, but commit is unnecessary because SELECT does not modify the database.

3. INSERT/UPDATE/DELETE REQUIRE COMMIT

BIGGEST EXAM QUESTION.
Without commit: cur.execute("INSERT INTO students VALUES (1,'Ali')")
The query executes INSIDE the transaction, but changes are not permanently saved.
Need: conn.commit()

4. SINGLE QUOTES VS DOUBLE QUOTES

Strings → SINGLE quotes.

CORRECT: INSERT INTO students VALUES (1, 'Ali')

WRONG: INSERT INTO students VALUES (1, "Ali")
In PostgreSQL: 'Ali' → string. "Ali" → identifier/column name.
PostgreSQL will throw: Error: column "Ali" does not exist.

5. TABLE NAME CASE SENSITIVITY

If a table is created as: CREATE TABLE Students (id INT);
PostgreSQL internally stores it as: students.

So SELECT * FROM students; works.
BUT: SELECT * FROM "Students"; fails unless the table was explicitly created with quotes.

6. PLACEHOLDERS IN PSYCOPG2

DO NOT DO THIS: (Dangerous + deducts marks)
cur.execute(f"SELECT * FROM students WHERE name='{name}'")

CORRECT:
cur.execute("SELECT * FROM students WHERE name=%s", (name,))

VERY IMPORTANT: (name,) NOT (name).
Because (name) is a string, not a tuple. Single-element tuples NEED a comma.

CRUD Operations Properly

CREATE (INSERT)

cur.execute(
    "INSERT INTO students(id,name,cgpa) VALUES(%s,%s,%s)",
    (1, "Ali", 3.5)
)
conn.commit()

READ (SELECT)

cur.execute("SELECT * FROM students")
rows = cur.fetchall()
for row in rows:
    print(row)
FunctionMeaning
fetchone()returns ONE row
fetchall()returns ALL rows
fetchmany(3)returns 3 rows

CURSOR EXHAUSTION

cur.execute("SELECT * FROM students")
print(cur.fetchone()) → Gives first row.
print(cur.fetchone()) → Gives second row.
The cursor moves forward automatically.

UPDATE

cur.execute(
    "UPDATE students SET cgpa=%s WHERE id=%s",
    (3.9, 1)
)
conn.commit()

MOST DANGEROUS MISTAKE: UPDATE students SET cgpa=4.0;
NO WHERE clause. Updates EVERY row.

DELETE

cur.execute(
    "DELETE FROM students WHERE id=%s",
    (1,)
)
conn.commit()

BIGGEST DELETE TRAP: DELETE FROM students;
Deletes ALL rows. The table still exists.

CommandRemoves rowsRemoves table structure
DELETEYESNO
TRUNCATEYES (all rows)NO
DROP TABLEYESYES

Transactions

  • ROLLBACK: conn.rollback() undoes changes since the last commit. No changes saved.
  • AUTOCOMMIT: Normally PostgreSQL uses transactions. Can enable: conn.autocommit = True. Then commit is unnecessary.

SQL Exam Trick Questions

  • Q: cur.execute("SELECT * FROM students"); conn.commit() - Will it execute?
    A: YES. Commit is unnecessary but valid.
  • Q: cur.execute("INSERT INTO students VALUES (1,'Ali')") - Will row permanently save?
    A: NO, not until commit.
  • Q: cur.execute("SELECT * FROM students WHERE id=%s", (1)) - Will it execute?
    A: NO properly. Because (1) is an integer, not a tuple. Need: (1,).
  • Q: SELECT name age FROM students; - Will it execute?
    A: YES. SQL interprets this as implicit alias: SELECT name AS age.
  • Q: SELECT * FROM students WHERE cgpa = NULL; - Will it work?
    A: NO. Need WHERE cgpa IS NULL. NULL is not comparable with =.
  • Q: DELETE * FROM students; - Will it execute?
    A: NO. Syntax error. Correct: DELETE FROM students;.
  • Q: UPDATE students cgpa=3.5 WHERE id=1; - Will it execute?
    A: NO. Missing SET. Correct: UPDATE students SET cgpa=3.5 WHERE id=1;.

PART 2 — Firebase Realtime Database CRUD

The completely different world of NoSQL. Firebase stores data as a JSON tree.

Structure & Connecting

import firebase_admin
from firebase_admin import credentials, db

cred = credentials.Certificate("serviceAccountKey.json")
firebase_admin.initialize_app(cred, {
    'databaseURL': 'https://your-project.firebaseio.com/'
})

CRUD Operations

CREATE (Insert Data)

ref = db.reference("students")
ref.child("1").set({
    "name": "Ali",
    "cgpa": 3.5
})

SQL vs Firebase Difference: SQL uses tables → rows → columns. Firebase uses a JSON tree. No tables, no rows, no joins.

READ

# Read all
ref = db.reference("students")
data = ref.get()
print(data)

# Read specific child
student = ref.child("1").get()

UPDATE

ref.child("1").update({
    "cgpa": 3.9
})

Only updates specified field.

DELETE

ref.child("1").delete()

Removes the node.

Firebase Theory Traps

1. set() OVERWRITES

ref.child("1").set({"name": "Ali"})
Replaces the ENTIRE node. Existing fields (like cgpa) are removed.

2. update() DOES NOT OVERWRITE

ref.child("1").update({"cgpa": 4.0})
Only updates cgpa, leaves name intact.

3. PUSH CREATES UNIQUE KEYS

ref.push({"name": "Ali"})
Firebase auto-generates a key (e.g., -Mxyz123).

Firebase Exam Questions

  • Q: Difference between set() and update()?
    A: set() overwrites the entire node. update() does partial modification.
  • Q: What happens if path does not exist? ref.child("100").get()
    A: Returns None.
  • Q: Can Firebase perform joins?
    A: NO. You must manually restructure/denormalize data.
  • Q: Difference between push() and set()?
    A: push() uses auto-key generation. set() uses an explicit key.
