// Section C: Code Challenges

Write real Verilog. Starts basic, ends with a Boss challenge: Counter + ALU wired together as one system. Use the textarea, then reveal the answer.

CODE · C01dataflow
Write a complete Verilog module for a 2-input XOR gate using dataflow modeling (assign statement). Name it xor_gate. Inputs: a, b. Output: y.
model answer
module xor_gate(
    output y,       // output first, convention
    input  a, b
);
    assign y = a ^ b;   // ^ is bitwise XOR
endmodule
Key points: Output declared first (convention). assign drives a wire — no need to say wire for output since it's the default for non-reg outputs. The expression a ^ b is continuously evaluated — any change in a or b instantly updates y.
CODE · C02behavioral
Write a behavioral half adder. Inputs: a, b. Outputs: sum, carry. Use an always block with @(*).
model answer
module half_adder(sum, carry, a, b);
    input      a, b;
    output reg sum, carry;  // reg because driven in always

    always @(*) // @(*) = trigger on any input change
    begin
        sum   = a ^ b;  // blocking = fine for combinational
        carry = a & b;
    end
endmodule
Why output reg? Because sum and carry are assigned inside an always block. Anything assigned inside always must be reg.

Why @(*) not @(posedge clk)? This is combinational logic — it has no clock. It must react instantly to input changes, not on a clock edge. @(*) ensures both a and b are in the sensitivity list automatically.

Why = not <=? Combinational blocks should use blocking =.
CODE · C03sequential
Write a D flip-flop with an asynchronous active-high reset. Inputs: D, clk, reset. Outputs: Q, Qbar. When reset is high, Q = 0 immediately (no clock needed).
model answer
module d_ff(Q, Qbar, D, clk, reset);
    input      D, clk, reset;
    output reg Q, Qbar;

    always @(posedge clk, posedge reset)
    begin
        if (reset) begin
            Q    <= 1'b0;
            Qbar <= 1'b1;
        end
        else begin
            Q    <= D;
            Qbar <= ~D;
        end
    end
endmodule
Async reset: posedge reset is in the sensitivity list alongside posedge clk. This means: run the block on a rising clock edge OR when reset goes high. The if (reset) check inside handles it — reset wins regardless of clock.

Sync reset would only have @(posedge clk) — reset only takes effect on a clock edge. Async is more common in practice.

Non-blocking <= is correct here — sequential logic.
CODE · C04sequential
Write a 32-bit up counter. On each positive clock edge it increments by 1. An active-high async reset sets it back to 0. Name the output count. Write its complete testbench in the same file.
module + testbench in one file
// ─────────── MODULE ───────────
module counter_32(count, clk, reset);
    input          clk, reset;
    output reg [31:0] count;

    always @(posedge clk, posedge reset)
    begin
        if (reset)
            count <= 32'd0;
        else
            count <= count + 1;
    end
endmodule

// ─────────── TESTBENCH ───────────
module counter_32_TB;  // no port list — testbenches never have ports
    reg          clk, reset;   // inputs to UUT → reg (you drive them)
    wire [31:0]  count;        // output of UUT → wire (it drives it)

    // Instantiate the design under test
    counter_32 uut (
        .count(count),
        .clk(clk),
        .reset(reset)
    );

    // Clock: toggle every 5 units → period = 10 time units
    always #5 clk = ~clk;

    initial begin
        clk   = 0;
        reset = 1;         // assert reset at start
        #12  reset = 0;    // release after 12 units
        #200 $finish;      // run for 200 units then stop
    end
endmodule
The key pattern for any testbench:
1. No port list on the testbench module itself.
2. UUT inputs → reg. UUT outputs → wire.
3. Instantiate with named connections .port(local_signal).
4. Separate always block for clock. initial block for stimulus.
5. End with $finish.

Why #12 for reset release? So the first clock edge hits while reset is still high — confirming reset works — then count starts from 0.
CODE · C05⚡ BOSS CHALLENGE
Design a two-module system in a single file:

Module 1 — op_counter: A 2-bit up counter that wraps around (0→1→2→3→0→...). Clock + async reset. Output: op[1:0].

Module 2 — alu_unit: A 4-bit ALU. Inputs: A[3:0], B[3:0], sel[1:0]. Output: result[4:0] (5 bits to hold carry). Operations: sel=0 → AND, sel=1 → OR, sel=2 → Addition, sel=3 → Subtraction.

Module 3 — counter_alu_top: The top-level. Instantiates both modules. The counter's output directly drives the ALU's sel — so each clock cycle the operation automatically changes.

Then write two separate testbenches in the same file: one tests the counter alone, one tests the whole top-level system.
complete solution — 5 modules in one file
// ═══════════════════════════════════════════
// MODULE 1: 2-bit up counter
// Wraps: 0→1→2→3→0→1→...
// ═══════════════════════════════════════════
module op_counter(op, clk, reset);
    input          clk, reset;
    output reg [1:0] op;

    always @(posedge clk, posedge reset)
    begin
        if (reset)
            op <= 2'b00;
        else
            op <= op + 1;  // 2-bit auto-wraps at 3→0
    end
endmodule

// ═══════════════════════════════════════════
// MODULE 2: 4-bit ALU
// sel=0:AND  sel=1:OR  sel=2:ADD  sel=3:SUB
// result is 5 bits to hold carry/borrow
// ═══════════════════════════════════════════
module alu_unit(result, A, B, sel);
    input      [3:0] A, B;
    input      [1:0] sel;
    output reg [4:0] result; // 5 bits: bit4 is carry/borrow

    always @(*)
    begin
        case (sel)
            2'b00: result = {1'b0, A & B};     // AND  — no carry
            2'b01: result = {1'b0, A | B};     // OR   — no carry
            2'b10: result = A + B;              // ADD  — carry in bit4
            2'b11: result = A - B;              // SUB  — borrow in bit4
            default: result = 5'b0;
        endcase
    end
endmodule

// ═══════════════════════════════════════════
// MODULE 3: TOP-LEVEL — wires counter into ALU
// counter.op → alu.sel
// Every clock edge, op increments → ALU op changes
// ═══════════════════════════════════════════
module counter_alu_top(result, A, B, clk, reset);
    input      [3:0] A, B;
    input              clk, reset;
    output     [4:0] result;

    wire [1:0] current_op;  // internal wire: counter → alu

    // Instance 1: counter generates the select signal
    op_counter ctr_inst (
        .op(current_op),
        .clk(clk),
        .reset(reset)
    );

    // Instance 2: ALU uses that select signal
    alu_unit alu_inst (
        .result(result),
        .A(A),
        .B(B),
        .sel(current_op)   // <— driven by counter output
    );
endmodule

// ═══════════════════════════════════════════
// TESTBENCH 1: Tests op_counter ALONE
// Verifies it counts 0→1→2→3→0→... and resets
// ═══════════════════════════════════════════
module tb_counter;
    reg          clk, reset;
    wire [1:0]  op;

    op_counter uut (.op(op), .clk(clk), .reset(reset));

    always #5 clk = ~clk;

    initial begin
        $monitor($time, " reset=%b op=%b", reset, op);
        clk = 0; reset = 1;
        #12 reset = 0;   // release reset
        #100 $finish;   // watch 10 clock cycles
    end
endmodule

// ═══════════════════════════════════════════
// TESTBENCH 2: Tests the FULL SYSTEM (top-level)
// Feeds fixed A and B, watches result change each cycle
// as counter cycles through AND→OR→ADD→SUB→AND→...
// ═══════════════════════════════════════════
module tb_system;
    reg  [3:0]  A, B;
    reg          clk, reset;
    wire [4:0]  result;

    counter_alu_top uut (
        .result(result),
        .A(A), .B(B),
        .clk(clk),
        .reset(reset)
    );

    always #5 clk = ~clk;

    initial begin
        $monitor($time, " A=%d B=%d result=%d", A, B, result);
        clk = 0; reset = 1;
        A = 4'd6; B = 4'd3;   // fixed inputs: 6 and 3
        // Expected cycle: AND=0010, OR=0111, ADD=01001, SUB=00011
        #12 reset = 0;
        #80 $finish;
    end
endmodule
How multiple testbenches work in one file: Each is just another module with no ports. When you simulate in Xilinx ISE, you right-click and select which module is the "top" for simulation. You can switch between tb_counter and tb_system — same file, different simulations.

The internal wire current_op: This is what connects two modules inside the top-level. It's declared as a wire — output of counter drives it, ALU input reads it. This is the core of structural composition in Verilog.

Why the ALU uses @(*) not @(posedge clk)? The ALU is combinational — it has no state, it just computes instantly based on inputs. The counter is what provides the clock-driven state. This separation of concerns is how real hardware is designed.

The counter auto-wraps because it's a 2-bit reg. When it hits 2'b11 (3) and you add 1, it overflows back to 2'b00 (0). The hardware handles this naturally.

// Section D: Multiple Testbenches Explained

A full breakdown of how to write and manage multiple testbenches — from basic to multi-module systems.

The Core Idea

A testbench is just a module with no ports. You can have as many of them as you want in a single file. In simulation (Xilinx ISE / ModelSim), you pick which module to simulate as "top." Each testbench is independent.
Pattern: multiple testbenches in one .v file
structure
// ── your actual modules ──────────────────────────────────────

module module_A(...); ... endmodule
module module_B(...); ... endmodule
module top_level(...); ... endmodule  // instantiates A and B

// ── testbench for module_A alone ─────────────────────────────
module tb_A;                   // no port list
    reg  input_to_A;
    wire output_of_A;
    module_A uut (.in(input_to_A), .out(output_of_A));
    initial begin
        // drive input_to_A, observe output_of_A
        #100 $finish;
    end
endmodule

// ── testbench for the full system ───────────────────────────
module tb_system;              // separate module, same file
    reg  sys_clk, sys_reset;
    wire sys_out;
    top_level uut (.clk(sys_clk), .reset(sys_reset), .out(sys_out));
    always #5 sys_clk = ~sys_clk;
    initial begin
        sys_clk = 0; sys_reset = 1;
        #20 sys_reset = 0;
        #200 $finish;
    end
endmodule
In Xilinx ISE: Set the "Sources for" dropdown to "Simulation." Right-click the testbench module you want to run and select it as top. You can switch between tb_A and tb_system without changing files.

Naming convention: Always name your testbench with a _TB or tb_ suffix. Makes it instantly obvious what's a design module vs a test module.
CONCEPT CHECK
When your testbench has two signals with the same name as ports in different modules you instantiate, how do you keep them separate?
answer
// You give the LOCAL signal a different name.
// The .port_name(local_signal) syntax separates them cleanly.

module tb_example;
    reg  my_clk, my_reset;      // LOCAL names in testbench
    wire [3:0] counter_out;
    wire [4:0] alu_out;

    // Both modules have a 'clk' port — but we connect them
    // to the SAME local signal (my_clk) just fine:
    op_counter c1 (.clk(my_clk), .reset(my_reset), .op(counter_out));
    alu_unit   a1 (.A(4'd5), .B(4'd3), .sel(counter_out[1:0]), .result(alu_out));

    always #5 my_clk = ~my_clk;
    initial begin
        my_clk = 0; my_reset = 1;
        #12 my_reset = 0;
        #80 $finish;
    end
endmodule
The key insight: .clk(my_clk) means "connect the module's port named clk to my local signal called my_clk." The local testbench signal namespace and the module port namespace are completely separate. You can name your local signals anything — only the port name (after the dot) must match the module definition.
$MONITOR vs $DISPLAY — when to use which
comparison
// $display: prints ONCE when the line executes — like printf in C
$display("A=%b B=%b result=%b", A, B, result);
// Use when you want to print at specific moments in your stimulus

// $monitor: prints AUTOMATICALLY whenever any listed signal changes
// Only ONE $monitor can be active at a time — last one wins
$monitor($time, " A=%d B=%d result=%d", A, B, result);
// Use at the start of initial block — it watches forever
// Great for sequential circuits where outputs change unpredictably

// Format strings:
// %b = binary   %d = decimal   %h = hex
// %t = time     $time gives current simulation time
Rule of thumb: Use $monitor for sequential/clocked designs where you want to see every output change. Use $display inside your stimulus sequence when you want to label specific test cases. Use both together — $display("--- TEST AND ---") then let $monitor print the actual values.
